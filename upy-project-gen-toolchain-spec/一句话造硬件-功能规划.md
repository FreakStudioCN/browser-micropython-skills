# 一句话造硬件 — AI 嵌入式生成工具功能规划

> 目标：用户只需用自然语言描述需求，系统自动完成从选型、生成代码、编译到烧录的全流程。

---

## 核心用户旅程

```
用户说一句话
    ↓
AI 理解意图 + 硬件选型
    ↓
自动生成代码 + 接线图
    ↓
库依赖自动解析安装
    ↓
一键编译 + 烧录
    ↓
串口调试 + 错误自动修复
```

## 整体架构

### 两大 Skill 体系

本项目的所有能力以 **Skill（SKILL.md）** 形式组织，存放在 `G:\MicroPython_Skills\`，共 25 个，分为两组：

```
G:\MicroPython_Skills\
├── 一句话造硬件流水线（10 个）          ← 本文件详细描述
│   ├── upy-analyze/         Phase 1  需求解析 + 驱动搜索
│   ├── upy-select-hw/       Phase 2  MCU 选型 + 引脚分配 + BOM
│   ├── upy-scaffold/        Phase 3  项目骨架生成
│   ├── upy-generate/        Phase 4  业务代码生成
│   ├── upy-simulate/        Phase 4.5 PC 端全流程模拟
│   ├── upy-deploy/          Phase 5  一键烧录运行
│   ├── upy-autofix/         Phase 6  编排协调层（分级决策 + 委托修复）
│   ├── upy-wiring/          Phase 7a 接线图生成
│   ├── upy-diagram/         Phase 7b 架构图 + 流程图
│   └── upy-gen-driver/     异常路径  冷门硬件驱动生成
│
├── 驱动开发规范化（15 个）              ← 详见 README.md
│   ├── 规范化：upy-norm-driver, upy-norm-main, upy-norm-pkg
│   ├── 生成：upy-gen-main, upy-gen-readme, upy-gen-pkg
│   ├── 优化：upy-opt-driver, upy-slim-driver
│   ├── 打包：upy-pack-driver
│   ├── 部署：upy-deploy-test
│   ├── 项目：upy-project
│   ├── 工具：upy-pkg-guide, fetch-doc
│   └── 设备：mpremote-device-interaction, mpremote-file-transfer, mpremote-live-session
│
└── upy-project-gen-toolchain-spec/    ← 本文件所在目录
    ├── 一句话造硬件-功能规划.md        整体架构 + 流程 + 模块设计
    ├── project-manifest.schema.json   各阶段数据契约
    ├── wiring.schema.json             接线图数据契约
    └── diagram.schema.json            架构图数据契约
```

### Skill 设计原则

**每个 Skill = SKILL.md（LLM 指令） + scripts/（确定性脚本）**：

| 层 | 职责 | 示例 |
|----|------|------|
| SKILL.md | LLM 角色定位 + 执行步骤 + 决策规则 + 约束 | 自然语言需求 → 器件清单 |
| scripts/*.py | 确定性操作（数据采集、格式转换、渲染） | triage.py 采集日志 + I2C 扫描 |

**Skill 间通过 project-manifest.json 传递状态**，每个 Phase 只写自己负责的字段（见 schema）。

### LLM vs 脚本分工原则

```
确定性、可复现、无判断力需求 → scripts/*.py
    ├── 日志解析（正则匹配）
    ├── I2C 硬件扫描（mpremote exec）
    ├── 数据格式转换（JSON → Markdown → Mermaid）
    ├── 文件渲染（SVG/PNG 生成）
    └── git 快照/回滚

需要判断力、上下文理解、创造性 → LLM（SKILL.md 指导）
    ├── 需求意图拆解
    ├── MCU 选型决策
    ├── 代码生成
    ├── 分级决策（autofix 委托哪个上游 skill）
    └── 场景设计（simulate 的数据发生器）
```

### 数据流

```
用户自然语言
    ↓
[Phase 1] upy-analyze ──→ project-manifest.json (phase: analyze)
    ↓
[Phase 2] upy-select-hw ──→ project-manifest.json (phase: select-hw, +mcu, +pinout, +bom)
    ↓
[Phase 3] upy-scaffold ──→ firmware/ 骨架 + project-manifest.json (phase: scaffold)
    ↓
[Phase 4] upy-generate ──→ firmware/ 完整代码 + project-manifest.json (phase: generate)
    ↓
[Phase 4.5] upy-simulate ──→ test/pc/sim_main.py（CLI+rich，PC 验证）
    ↓
[Phase 5] upy-deploy ──→ 设备运行 + 输出判定
    ↓                    ↓ FAIL
    PASS                  [Phase 6] upy-autofix ──→ 分级决策 → 委托上游 skill → 重新 deploy
    ↓
[Phase 7] upy-wiring + upy-diagram（可并行）
```

## 功能模块

### 1. 自然语言理解（输入层）
- 用户用一句话描述项目需求（中英文均支持）
- AI 拆解意图：功能需求 / 传感器 / 通信方式 / 触发条件
- 支持追问澄清（多轮对话补全信息）
- 示例：`"做一个温湿度监测仪，超过阈值发微信通知"`

### 2. 硬件选型推荐
- 根据需求自动推荐开发板型号（ESP32 / Arduino Uno / STM32 等）
- 列出所需传感器/模块清单（含型号、数量）
- 给出购买建议（价格区间、替代方案）
- 考虑用户已有硬件（可告知已有设备）

### 3. 接线图生成
- 自动生成引脚连接图（哪个引脚接哪里）
- 可视化接线示意图（Fritzing 风格或文字表格）
- 标注注意事项（电压、电流限制等）

### 4. 代码生成
- 支持多语言：Arduino C++ / MicroPython / ESP-IDF
- 生成完整可运行项目（含 main 文件 + 配置文件）
- 代码含必要注释，适合入门阅读
- 支持 Blockly 积木与代码双向同步（可选）

### 5. 库依赖自动管理（关键难点）
- 自动识别代码所需的第三方库
- 自动下载安装正确版本（避免版本冲突）
- 项目级依赖隔离（不同项目互不干扰）
- 参考：AutoEmbed 的 component-aware library resolution

### 6. 编译与烧录
- 内置编译器 或 云端编译（缩短本地环境配置时间）
- 自动检测串口 / COM 口
- 一键烧录到设备
- 编译错误自动 AI 解释 + 修复建议

### 7. 串口调试与反馈
- 实时串口监视器（日志输出）
- AI 解读设备输出（异常自动报警）
- 代码自动迭代修复循环（生成→测试→修复）

### 8. AI × Git 版本控制

- 项目创建时自动 `git init`，无需用户手动操作
- 每次 AI 生成/修改代码后自动提交，生成规范描述：
  - `feat: 添加 I2S 麦克风驱动`
  - `fix: 修复屏幕显示 BUG`
  - `refactor: 优化 WiFi 重连逻辑`
- 手动修改代码后，AI 自动识别变更内容并生成提交描述
- 烧录失败 / 设备异常时，一键回滚到上一个可用版本
- 对用户隐藏 Git 复杂性，以**时间轴 UI** 展示版本历史（类似 Figma 版本历史）
- 版本历史 = 项目开发日志，方便分享、复现、协作

> 差异化：所有竞品均无此功能，是入门创客"改乱了回不去"痛点的直接解法。

### 9. 项目模板库
- 常见场景模板（智能家居 / 环境监测 / 机器人 / 显示屏 / 网络通信）
- 一键复用 + 自定义修改
- 社区共享模板

### 功能 vs 形式适配

| 功能           |   网页   | 桌面 IDE | VS Code 插件 |
| ------------ | :----: | :----: | :--------: |
| 自然语言输入       |   ✅    |   ✅    |     ✅      |
| 硬件选型推荐       |   ✅    |   ✅    |     ✅      |
| 接线图生成        |   ✅    |   ✅    |     受限     |
| 库自动管理（upypi） | ⚠️ 需后端 | ✅ 本地执行 |     ✅      |
| 一键编译/烧录      | ❌ 串口受限 |   ✅    |     ✅      |
| mpremote 调试  |   ❌    |   ✅    |     ✅      |
| 错误自动修复       |   ✅    |   ✅    |     ✅      |

### 结论：桌面 IDE 为核心，网页+插件为入口漏斗

```
桌面 IDE（核心，跑完整链路）
    ↑ 引流
网页（模板库 + 在线体验 + SEO）
    ↑ 引流
VS Code 插件（覆盖已有开发者用户）
```

## 用户入口策略

**第一入口：首屏一个输入框**
产品打开就是 `"你想做什么？"` — 不选板子、不注册、直接说需求。零门槛冷启动，类似 ChatGPT 体验。

**第二入口：网页项目库**
后续项目库网页天然是 SEO 入口。用户搜"ESP32 温湿度"→ 落地页展示项目 → 引导"在 IDE 中打开" → 触发下载安装。

**第三入口：VS Code 插件（轻量钩子）**
不做完整功能，只做 AI 生成代码 + 跳转到桌面 IDE 烧录。覆盖已有开发者，降低迁移成本。

- **AI 核心**：LLM（Claude / GPT）+ RAG（硬件库知识库）

- **前端**：Electron 桌面应用
- **串口通信**：serialport（Node.js）

Phase 4:   upy-generate                   ← 业务代码生成（核心价值）

## VS Code 插件架构详细设计

### 整体架构

```
用户操作（VS Code 插件）
    │
    ├── 本地执行（插件直接调用）
    │     ├── mpremote → 串口烧录/调试（物理串口，服务器够不到）
    │     ├── git      → 版本控制（操作本地仓库）
    │     └── 文件读写 → 项目文件（VS Code 工作区）
    │
    └── 服务器 API 调用（其余全部）
          ├── /analyze     → Phase 1 需求解析 + upypi/GitHub 搜索（LLM）
          ├── /select-hw   → Phase 2 硬件选型推荐（引脚 mapping 服务器维护）
          ├── /scaffold    → Phase 3 项目骨架生成（模板填充）
          ├── /generate    → Phase 4 业务代码生成 + 语法检查 + 测试生成（LLM，已内化原 Phase 5/6）
          ├── /fix-error   → Phase 6 错误修复分析（LLM）
          ├── /gen-driver → 异常路径 PDF/Arduino → MPY 驱动转换
          └── /git-msg     → 提交描述生成（diff → LLM）
```

**引脚 mapping 放服务器的好处**：新开发板支持无需用户更新插件，服务器更新即生效。

### VS Code 插件技术可行性

插件本质是 Node.js 进程，以下能力均支持：

| 能力                  | API                            |
| ------------------- | ------------------------------ |
| 调用本地 mpremote / git | `child_process.spawn`          |
| 调用服务器 API           | `fetch` / `axios`              |
| 渲染接线图、时间轴、对话界面      | `WebviewPanel`                 |
| 读写本地项目文件            | `vscode.workspace.fs`          |
| 显示 mpremote 实时输出    | `vscode.window.createTerminal` |

**插件只是本地执行层 + 服务器 API 的调用壳，轻量，业务逻辑全在服务器。**

### 模块一：需求分析 + 硬件选型（Phase 1-2）

**Phase 1 — upy-analyze：自然语言 → 器件清单 + 驱动搜索**

```
用户一句话
  → LLM 拆解意图：功能需求 / 传感器 / 通信方式 / 触发条件

  → upypi 无 → GitHub Search API fallback → 搜到 Arduino 代码则标记

  → 输出：BOM 表（型号/数量/单价/备注）+ 引脚分配表
```

**执行位置：** LLM 调用、upypi/GitHub 搜索 → 服务器；引脚数据库可本地缓存

### 模块五-B：PC 端全流程模拟（Phase 4.5）

**Phase 4.5 — upy-simulate：无需硬件的业务逻辑验证**

> 这是代码生成 skill，不是运行框架。LLM 读取 firmware/ 全部代码后，自主设计调度方案、Mock 数据发生器、可视化形式，生成 `test/pc/sim_main.py`。

```
Step 1: LLM 全量读取 firmware/ 上下文
  → main.py（回调包装 + 任务注册 + 数据流）
  → tasks/*.py（函数签名 + 业务逻辑分支）
  → drivers/*/mock.py（Mock __init__ 参数 + _raise_on 注入点）
  → conf.py（阈值 + 采样率 + 刷新间隔）
  → lib/scheduler/*.py（Scheduler API）
  → project-manifest.json（mode: timer|async|thread）

Step 2: LLM 自主设计 4 项决策
  调度方案：timer → time.sleep 循环 | async → asyncio.run | thread → threading
  Mock 数据发生器：gen_xxx(tick) 函数，数据随时间变化（核心——禁止静态常量）
  可视化：CLI+rich 优先（所有项目默认）| tkinter GUI 可选实验模式
  数据场景：normal / temp_rising / temp_dropping / intermittent_failure / sensor_death

Step 3: 生成 test/pc/sim_main.py（+ timer 模式下 sim_scheduler.py）
  → flake8 + pylint 校验 → 循环修复直到通过
  → AskUserQuestion 询问是否运行

Step 4: 运行 + 判定
  → python test/pc/sim_main.py --ticks 30 --mode gui --scenario temp_rising
  → 观察：告警是否在温度跨越阈值时触发？传感器故障后是否恢复？
  → PASS → 进入 deploy
  → FAIL → 修复 sim_main.py（模拟脚本本身的 bug），不修改 firmware/
```

**核心约束：**
- **不修改 firmware/ 下任何文件**（除非发现确定的 bug）
- Mock 数据必须用发生器函数 `gen_xxx(tick)`，禁止静态常量
- 有 display 器件的项目用 rich Panel 模拟虚拟屏幕
- 可被 upy-autofix 调用：修复后先 PC 验证（2-3s），再决定是否烧录

**执行位置：** 全部本地（CPython 运行，无硬件依赖）

### 模块六：自动化调试闭环（Phase 6）

**Phase 6 — upy-autofix：编排协调层（非独立修复机）**

核心定位：autofix 不自己写修复代码、不自己执行部署。它是轻量编排层——解析错误 → 分级决策 → 委托上游 skill 修复 → 验证。

```
Step 1: 读取日志（autofix 独有能力）
  → python tools/read_device_log.py --port {COM}
  → 设备端 rotating_logger 写入 /log/run_*.log → PC 端脚本读取合并

Step 2: 解析错误 + 硬件检测（autofix 独有能力）
  → 解析优先级 P0→P3：
    P0: Traceback / __MPY_ERR__:（Python 运行时异常）
    P1: rst cause:4 / Guru Meditation Error（硬件复位/内核 panic）
    P2: [FAIL] xxx / MemoryError / ENOMEM（运行时错误/内存不足）
    P3: 超时无输出（死循环/crash）
  → 硬件检测（软件修复之前必须先做）：
    对 I2C 设备 → mpremote exec i2c.scan()
    ├─ 有设备 → 软件问题，进入 Step 3 分级决策
    └─ 空 → software I2C → 降低速率 → 仍空 → 硬件问题 → 输出排查指引，不修代码

Step 3: 分级决策（autofix 独有能力——决定回退到哪个上游 skill）

  错误根因分层          →  委托 skill         →  修复方式
  ─────────────────────────────────────────────────────
  P0: 变量名拼错/缺少 import → autofix 直接改文件   →  简单字符串替换（不改架构）
  P0: 驱动 API 调用错误    → upy-generate      →  注入 error context，重生成 task 代码
  P1: I2C 地址冲突/引脚错   → upy-select-hw     →  重新分配地址/引脚
  P1: 看门狗/内核 panic    → upy-generate      →  修改调度策略（加 wdt.feed()、减 sleep）
  P1: 内存不足            → upy-generate      →  减小 buffer、gc.collect()、bytearray 复用
  P2: 传感器一直返回异常    → upy-generate      →  修改错误处理/重试逻辑
  P3: 死循环/无输出        → upy-generate      →  分析 main.py 执行流，修正阻塞点
  硬件: I2C 扫描为空       → 输出排查指引       →  接线/供电/上拉电阻/传感器/冲突（5 步）
  需求: 缺传感器/选错 MCU   → upy-analyze       →  告知用户需要重新分析需求

Step 4: 修复循环（最多 3 次尝试）

  attempt = 1..3:
    1. git commit 保存快照（autofix 独有——回滚安全保障）
    2. 委托上游 skill 修复（autofix 打包 error context → 上游 skill 产出修正代码）
    3. 可选：upy-simulate PC 端快速验证（2-3s，省去串口烧录延迟）
    4. 委托 upy-deploy 重新烧录运行
    5. 采集结果
    ├─ PASS → 成功退出，记录修复历史
    └─ FAIL → 重新 Step 3 分级决策（可能升级回退层级）

Step 5: 3 次失败 → 放弃 + 回滚 + 通知用户

  → git checkout 回滚到修复前版本
  → 输出卡点总结：错误类型 + 3 次尝试内容 + 失败原因
  → 建议手动排查方向
  → 记录到 logs/error_report.json（错误回流）

Step 6: 错误数据回流（autofix 独有——驱动 CI/CD 持续改进）

  → logs/error_report.json 记录错误类型/traceback/修复尝试/最终结果
  → 定期聚合分析 → 发现 skill.md 知识盲区
  → 反馈到 upy-autofix / upy-generate / upy-select-hw 的 SKILL.md
```

**autofix 的自有代码量极小：** 核心逻辑是 `triage.py`（356行，错误解析 + I2C 硬件检测 + git 快照/回滚 + 分级决策 + 循环控制），修复和部署委托给上游 skill，不重复实现。

**执行位置：** mpremote 硬件检测 → 本地；LLM 分级决策 → 服务器；委托上游 skill → 服务器（generate/select-hw/analyze）或本地（deploy/simulate）

### 模块七：冷门硬件支持（异常路径 — upy-gen-driver）

当 upypi 和 GitHub 都找不到驱动时的异常处理。整合 PDF→MPY 和 Arduino→MPY 两条路径，输入来源不同但流程一致：**提取 → 生成调试版 → 硬件验证 → 脱调试 → 规范化**。

**输入来源 → 统一处理流程：**

| 输入来源                 | 提取方式                                     | 硬件验证 | 规范化             |
| -------------------- | ---------------------------------------- | ---- | --------------- |
| 用户提供 PDF 数据手册        | `scripts/extract_pdf.py`（pymupdf）→ 提取通信协议/寄存器表/初始化序列 | `mpremote resume run` 最多 10 轮 | upy-norm-driver |
| 用户提供 Arduino/C++ 代码  | `scripts/convert_arduino.py` → API 映射（digitalWrite→Pin, Wire→I2C, SPI→SPI） | 同上 | upy-norm-driver |
| GitHub 找到 Arduino 代码 | clone → 分析 → 同上                          | 同上 | upy-norm-driver |
| 仅提供芯片型号              | 搜索 datasheet PDF → 下载 → 同 PDF 流程         | 同上 | upy-norm-driver |

**冷门硬件完整路径：**
```
upy-analyze 搜 upypi + GitHub → 无结果
  ↓
用户提供 PDF / Arduino 代码 / 芯片型号
  ↓
upy-gen-driver:
  ├─ PDF → extract_pdf.py → LLM 提取协议/寄存器
  ├─ Arduino → convert_arduino.py → API 映射
  ├─ LLM 生成调试版驱动（含全量自检：寄存器写读验证、AT 指令回显、初始化分步日志）
  ├─ 硬件验证循环（最多 10 轮）：
  │     mpremote resume run 调试版驱动 → 捕获输出 →
  │     LLM 分析输出 PASS/FAIL → 编辑驱动 → 再次验证
  │     （resume run 走 REPL 不写 Flash，快速迭代）
  ├─ 验证通过 → 脱调试代码（剥离自检逻辑 + 日志点）
  ├─ upy-norm-driver 规范化
  └─ 输出到 firmware/drivers/{chip}_driver/
  ↓
→ 回到主流程 Phase 4（upy-generate）
```

**强约束：**
- 驱动必须符合 upy-norm-driver 全部 P0 规则（依赖注入强制项）
- PDF 提取的寄存器表必须保留原文引用（第几页、哪个表）
- Arduino 映射不能机械翻译，需理解原代码逻辑

**执行位置：** PDF 解析、API 映射、LLM 驱动生成 → 服务器；硬件验证（mpremote resume run）→ 本地；脱调试 + 规范化 → 服务器

## AI 生成嵌入式代码：MicroPython vs Arduino 平台对比

> 从 AI 辅助开发的角度，分析哪个平台更适合作为"一句话造硬件"的代码生成目标。

### 评估维度

#### 1. 工程化调试闭环

**MicroPython 胜**

- REPL 交互式调试：无需重新烧录，直接在设备上执行单行代码验证
- `mpremote` 工具链：`run`、`exec`、`fs` 命令支持热更新单文件
- AI 生成代码 → `mpremote run main.py` → 立即看结果，闭环极短
- Arduino：每次改动必须完整编译（10s~60s）+ 烧录，闭环长

#### 2. 自动化测试框架

**MicroPython 胜**

- `unittest` 模块可直接在设备上运行
- AI 可生成引脚 mock、外设 stub，结构与 PC 端 Python 测试一致
- Arduino 没有标准测试框架，`ArduinoUnit` 等第三方库集成复杂
- AI 对 Python 测试代码的生成质量远高于 C++ 测试代码（训练数据优势）

#### 3. 开发者感知（算力/帧率/外设调度）

**Arduino 略胜，但差距在缩小**

- Arduino：`micros()`/`millis()` 精度高，裸机调度透明，资源占用可精确计算
- MicroPython：GC 暂停、解释器开销不透明，但：
  - `utime.ticks_us()` 可测帧率
  - `micropython.mem_info()` 可查内存
  - AI 可生成标准化 profiling 模板，弥补感知缺口

#### 4. 基本开发技能（内存/看门狗/重连）

| 能力   | MicroPython                | Arduino               |
| ---- | -------------------------- | --------------------- |
| 内存管理 | `gc.collect()` 显式调用，AI 易生成 | 手动 malloc/free，AI 易出错 |
| 看门狗  | `machine.WDT()` 一行搞定       | 寄存器操作，平台差异大           |
| 重连机制 | `try/except` 结构清晰，AI 生成质量高 | 状态机写法，AI 容易漏边界        |
| 异常保护 | Python 异常体系完整              | 需手写错误码，AI 容易不一致       |

MicroPython 的异常/重连代码，AI 生成的可靠性更高。

#### 5. 生成流程图/架构图/依赖图

**MicroPython 胜**

- Python 模块结构（`import`、类、函数）对静态分析友好
- AI 可直接从 `.py` 文件提取调用图、依赖图
- Arduino `.ino` 文件的隐式函数声明、多文件拼接机制让依赖分析困难
- Mermaid/PlantUML 生成：Python 代码结构更线性，AI 转换准确率更高

### 结论

| 维度     | 胜者          |
| ------ | ----------- |
| 调试闭环   | MicroPython |
| 自动化测试  | MicroPython |
| 运行感知   | Arduino（微弱） |
| 基本开发技能 | MicroPython |
| 架构可视化  | MicroPython |

**对 AI 辅助嵌入式开发，MicroPython 是更适合的目标平台。** 核心原因是 Python 的结构化特性让 AI 生成、测试、分析代码的质量和可靠性系统性地高于 C/C++，而不只是"库多少"的问题。

### upypi 接口使用规范

upypi 是结构化驱动包仓库，直接调用 API，无需本地向量库。

```
# 搜索驱动包
GET https://upypi.net/api/search?q={关键词}
→ { query, results: [{name, url}] }

# 获取包详情（兼容性 + 文件列表）
GET https://upypi.net/pkgs/{name}/{version}/package.json
→ { name, version, chips, fw, urls }

# 读取驱动文档（规范化 README）
GET https://upypi.net/pkgs/{name}/{version}/README.md

# 安装到设备
mpremote mip install https://upypi.net/pkgs/{name}/{version}/package.json
```

**驱动包标准目录结构：**
```
{chip}_driver/
├── code/
│   ├── {chip}.py
│   ├── main.py
│   └── {subpkg}/
├── package.json
├── README.md
└── LICENSE
```

### 用户报错数据回流机制

**核心价值：** 用户真实失败案例是发现知识盲区的最直接信号，驱动 skill.md、error-patterns、board-mapping 持续迭代。

```
用户报错 → /fix-error 处理
    ↓
服务器埋点记录（可选上报，用户授权）：
  ├── 错误类型 + 原文 traceback
  ├── 开发板型号 + upypi 包名/版本
  ├── AI 第几次修复成功 / 最终失败
  └── 当前使用的 skill.md 版本号
```

**数据分析流程（每周/月定期）：**

```
聚合报错数据
    ↓
按错误类型分类：
  ├── ImportError     → upypi 缺包 / 包名不一致
  ├── OSError         → 接线错误 / board-mapping 有误
  ├── AttributeError  → API 用法与 README 不符
  └── 3次修复失败     → skill.md 知识盲区（最高优先级）
    ↓
LLM 聚合分析 → 输出改进建议报告
    ↓
人工审核确认
    ↓
更新 skill.md / error-patterns / board-mapping
    ↓
同一套 CI/CD 流提交 → 服务器热重载
```

**错误信号与知识库的映射关系：**

| 错误类型                  | 说明                    | 更新目标                           |
| --------------------- | --------------------- | ------------------------------ |
| `ImportError` 高频      | upypi 缺少某类驱动          | 通知 upypi 补充包 / upy-select-hw   |
| `OSError: [Errno 19]` | 引脚 mapping 错误         | board-mapping 数据库              |
| `AttributeError`      | README API 示例与实际不符    | upy-generate / upy-norm-driver |
| 3次修复全部失败              | 当前 autofix skill 知识盲区 | upy-autofix SKILL.md           |
| 特定开发板高频报错             | 该板 mapping 或兼容性有问题    | board-mapping + upy-wiring     |

**隐私原则：**
- 只收集错误类型和技术上下文，不收集用户业务代码
- 插件内明确提示，用户勾选"帮助改进产品"后才上报
- 数据匿名化处理，不关联用户身份
